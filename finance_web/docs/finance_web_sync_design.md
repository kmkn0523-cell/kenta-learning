## 1. ゴールと前提

### ゴール
- finance_web（家計管理アプリ）を **複数の端末（スマホ・PC・タブレット）** で同じデータを使えるようにする
- マネタイズの軸として **¥500/月のサブスク** に組み込めるレベルの完成度

### 前提
- 現状はローカル localStorage のみ。データは **暗号化済JSON**（パスワード派生鍵で復号）
- 既存バックアップ機能: 端末A→JSON書き出し→端末Bで読み込みの **手動同期** は実装済み
- セキュリティ要件: **クラウド側で平文を扱わない**（E2E暗号化を維持）

| D. iCloud Drive / Dropbox | $0 | 2〜3日 | OS依存（PWAだと制限あり） | E2E維持 | ⭐⭐ |

### 推奨: **案C（Vercel KV）**

理由:
1. **既存の暗号化スキーマがそのまま使える** — KVには暗号化済み文字列を保存するだけ
2. **Vercel Marketplace 経由で1クリック導入** — env vars 自動設定
3. **無料枠が個人利用で十分** — 日10K request、4000ユーザー収容可能
4. **マネタイズと相性◎** — 既存 Vercel デプロイ環境にそのまま乗る

## 3. アーキテクチャ

```
端末A ──[暗号化済JSON]──▶ Vercel API Route (Edge Function) ──▶ Vercel KV
                                                  │
端末B ◀──[暗号化済JSON]────────────────────────────┘
       (起動時pull + 変更時push + 30秒ごとpoll)
```

### データフロー

| トリガー | 動作 |
|---|---|
| アプリ起動 | サーバ version をpullしてローカルと比較→必要なら復元 |
| データ変更 | 5秒debounceで自動push |
| 30秒ポーリング | アクティブタブのみ、他端末の更新を取りに行く |
| 競合検出 | UI出して「サーバ版/ローカル版/マージ」を選ばせる |

---

## 4. セキュリティ設計

### 鍵管理
- パスワードは **サーバに送らない**（既存と同じ）
- ユーザーID = `SHA-256(salt + password)` のハッシュ
- サーバはこのハッシュだけ知る。逆算でパスワードは復元不可

### データ
- KV保存値 = 既存の暗号化済JSON（中身はサーバから見えない）
- 既存の `ENC_SALT_KEY` を含めて保存する → 別端末でも同じパスワードで復号可

### 既存仕様との整合
- **パスワード忘れたら全データ復元不可** → 既存と同じ仕様
- 既存のバックアップJSON機能は残す（オフライン保険用）

## 5. API 仕様

### `GET /api/sync?userId={hash}`
- レスポンス: `{ version: number, data: string, updatedAt: string } | null`
- データなし（初回）→ `null`

### `PUT /api/sync`
```json
{
  "userId": "abc123...",
  "version": 5,
  "data": "暗号化されたJSON文字列"
}
```
- サーバ側 version === クライアント version → version+1 で保存、200 OK
- サーバ側 version !== クライアント version → **409 Conflict** + 最新データを返す

### 競合解決UI

```
┌─────────────────────────────────┐
│ ⚠️ 他端末で更新がありました          │
│                                  │
│ ローカル: 2026-05-19 21:30       │
│ サーバ:   2026-05-19 22:15       │
│                                  │
│ [サーバ版を採用]                  │
│ [ローカル版で上書き]              │
│ [バックアップしてサーバ版を採用]   │
└─────────────────────────────────┘
```

マージは家計データの性質上難しいので **3択UI** とする。

## 11. 別案メモ（採用しない理由）

### 案A: Vercel Blob
- 楽観ロックを自前実装する必要あり（KVは Redis SETNX が使える）
- ファイルストレージなので競合制御が原始的

### 案B: Supabase
- Postgres にスキーマを切る → カテゴリやカラム構造がサーバに漏れる
- E2E暗号化を維持しようとすると Postgres の利点を捨てることに
- 工数も最大級

### 案D: iCloud Drive / Dropbox
- PWA からの直接アクセスに制約が多い
- 「家計」のためだけにクラウドストレージを使わせるUXが悪い

### 案E: Google Drive API
- 認証フローが複雑（OAuth）
- Vercel KV より工数大、コストはほぼ同じ
