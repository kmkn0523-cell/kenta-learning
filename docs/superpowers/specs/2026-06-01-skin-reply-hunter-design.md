# skin リプ巡回ツール `skin_reply_hunter.py` 設計

- 日付: 2026-06-01
- ステータス: 設計合意（実装プラン作成前）
- 関連: `docs/superpowers/specs/2026-05-18-skin-monetize-roadmap-design.md`（半年マネタイズ・ロードマップ）

---

## 1. 背景と目的

skinアカウントの半年マネタイズ計画（2026/11/17までに月3〜5万円・Threads主軸）で、集客の柱は「能動リプ運用」——他人の投稿に価値あるリプをして認知とフォロワーを増やすこと。

ところが `skin/skin_daily_log.md` を見ると、5/18以降のリプ件数はほぼゼロで運用が止まっている。原因は能動リプ最大の摩擦＝「リプ先を毎日探す手間」。**探す → 書く → 記録** がバラバラの作業になっていて続かない。

このツールは、その3つを1コマンドにまとめ、毎日ラクに回せるようにする「半自動ツール」。

## 2. なぜ「半自動」なのか（Threads APIの制約）

技術調査の結論:

- 「テーマで他人の公開投稿を自動収集する」には Threads Graph API の `threads_keyword_search` 権限が必要で、これは Meta社の **App Review**（ビジネス認証を伴う重い審査）が必須。保留中のInstagram insights申請よりさらに重い。
- 特定アカウント（`skin_account_targets.json` の10人）の投稿を **名前指定で自動取得する標準APIは存在しない**。

→ フル自動の「リプ先自動収集」は今の環境では非現実的。現実解は **「探しに行く動作を、ツールが徹底的に楽にする」半自動**。API審査ゼロで今日から動く。

## 3. スコープ

**やること:**

- `skin_account_targets.json` の10アカウントを「テーマ・絡みどころメモ・前回リプ日」付きで一覧表示し、順番にブラウザで開いて巡回する
- 見つけた投稿の文章を貼ると、内容に合う下書きを3案提示する（既存テンプレのキーワード絞り込み）
- リプ件数を記録し、`skin_daily_log.md` に自動追記、アカウント別の履歴を JSON で管理する

**やらないこと（YAGNI）:**

- API経由の自動リプ投稿（スパム判定リスク・追加権限。投稿は手動コピペ）
- LLM / Claude API での下書き生成（新ライブラリ・コスト・承認が必要）
- 投稿文の具体語をテンプレに穴埋めする「カスタム化」
- 他人の公開投稿の自動収集（App Review必須）

## 4. 投稿方法の基本方針

下書き生成まではツールがやり、**最後の投稿はKentaが手動でThreadsアプリにコピペ**する。理由:

1. 他人の投稿へ機械的にAPI連投するとThreadsの**スパム判定リスク**がある（`skin_comment_helper.py` 自身が「やりすぎはスパム」と警告）
2. 投稿前にKentaが目視確認でき、**関係構築の質**を守れる
3. 自動リプ用の追加権限・実装が不要で**今日から動く**

→ **このツールはThreads APIを一切叩かない**。つまりネットワーク失敗・トークン切れの心配がなく、守るのはファイル読み書きと入力ミスだけ。

## 5. 全体像（1日の流れ）

```
起動
 ↓
① skin_account_targets.json の10アカウントを一覧表示
   （テーマ・絡みどころメモ・前回リプ日付き／巡回順に並べる）
 ↓
② 1人選ぶ → そのThreadsをブラウザで開く
 ↓
③ 良い投稿を見つけたら、その投稿の文章を貼り付ける
 ↓
④ ツールが話題をキーワード判定 → 合った下書きを3案提示
 ↓
⑤ 1案選ぶ → コピー用に表示（Kentaがアプリに貼って投稿）
 ↓
⑥ 「リプした」と記録 → カウント+1
 ↓（②〜⑥を繰り返し）
終了時：その日のリプ件数を skin_daily_log.md に自動追記
```

## 6. コンポーネント（小さな関数に分割）

| 関数 | 役割 | 依存 | テスト対象 |
|---|---|---|---|
| `load_targets()` | `skin_account_targets.json` を読む | json・ファイル | — |
| `load_history()` | `skin_reply_history.json` を読む（無ければ空） | json・ファイル | — |
| `sort_round_list(targets, history)` | 巡回順に並べる（未リプ最優先 → 古い順） | なし（純粋関数） | ◯ |
| `show_round_list(...)` | 巡回リストを画面表示 | `sort_round_list` | — |
| `open_account(url)` | ブラウザで開く（フォールバック付き） | webbrowser・subprocess | — |
| `suggest_drafts(post_text)` | 投稿文 → 下書き3案 | テンプレ（純粋関数） | ◯ |
| `record_reply(history, handle, today)` | 履歴を更新して返す | なし（純粋関数） | ◯ |
| `save_history(history)` | `skin_reply_history.json` に書き込む | json・ファイル | — |
| `append_daily_log(count, today)` | `skin_daily_log.md` に追記 | ファイル | ◯（ロジック部） |
| `main()` | 対話ループ全体 | 上記すべて | — |

**設計方針**: 「入力 → 出力」がはっきりした純粋ロジック（`sort_round_list` / `suggest_drafts` / `record_reply` / ログ追記ロジック）と、ファイル操作・ブラウザ・対話（`main` 周り）を分ける。テストは前者に集中する。

**テンプレの共通化**: いま `skin_comment_helper.py` の中にある `COMMENT_TEMPLATES`（6カテゴリ・各5〜7文）を、共通ファイル `skin_comment_templates.py` に切り出し、`comment_helper` と新ツールの両方から `import` する。テンプレ修正が1箇所で済み、`comment_helper` 側は読み込み行を1行差し替えるだけで動作は変わらない。

## 7. データファイル

**(a) `skin_account_targets.json`**（既存・読むだけ）
10アカウントの handle / url / theme / notes を持つ。

**(b) `skin_reply_history.json`**（新規・読み書き）
`progress.json` 系とは無関係で手動編集OK。日付は全て ISO `YYYY-MM-DD`。
```json
{
  "version": "1.0",
  "updated_at": "2026-06-01",
  "history": {
    "@shiori_wellness_kitchen": { "last_replied_at": "2026-05-30", "total_replies": 3 },
    "@lii_gohan":               { "last_replied_at": "2026-05-28", "total_replies": 1 }
  }
}
```

**(c) `skin_daily_log.md`**（既存・追記）
その日の枠に「リプ件数」だけ書き込む（後述）。

## 8. 下書き生成の仕様（`suggest_drafts`）

`skin_reply_helper.py` の `suggest_replies` と同じ手法（キーワードで話題を判定 → そのカテゴリのテンプレから候補を出す）。

**キーワード → カテゴリ**（`skin_comment_templates.py` の6カテゴリ）:
```
腸・便秘・発酵・食物繊維・腸活      → ①腸活・食事
寝・睡眠・夜更かし・寝不足          → ②睡眠・生活習慣
化粧水・保湿・乾燥・スキンケア      → ③スキンケア
ニキビ・肌荒れ・吹き出物・赤み      → ④ニキビ共感
サプリ・ビタミン・亜鉛・鉄・栄養    → ⑤サプリ・栄養
ファスティング・断食・16時間        → ⑥ファスティング
```

**判定が割れたとき:**

- 複数カテゴリに該当（例「腸活で肌が変わった」）→ **当たった数が多いカテゴリ**を優先。当たり数が同じなら **カテゴリ番号の若い方**（①→⑥の順）を選ぶ（結果が毎回同じになるよう順序を固定）
- どれにも該当しない → 「共感＋ひとこと質問」の**汎用テンプレ**を出し、「番号でカテゴリを手動選択もできる」と促す（取りこぼしゼロ）

各カテゴリのテンプレ（5〜7文）から `random.sample` で3案を抽出。

## 9. 巡回と記録

**(1) 巡回順**: `skin_reply_history.json` の `last_replied_at` を見て、**しばらくリプってない人を上に並べる**（未リプは「未」表示で最優先）。10アカウントをまんべんなく回せる。未リプが複数いるとき・最終リプ日が同じときは、`skin_account_targets.json` の元の並び順を保つ（安定ソート＝順序が毎回同じになる）。

**(2) ブラウザの開き方**（WSL2対応・新ライブラリなし）:
```
webbrowser.open(url) を試す
 ↓ ダメなら
cmd.exe /c start <url>（Windows側で開く）にフォールバック
 ↓ それも無理なら
URLを画面に表示（Kentaがクリック/コピペ）
```
`webbrowser` も `subprocess` もPython標準ライブラリ。最低でもURLは必ず表示されるので詰まらない。

**(3) `skin_daily_log.md` への自動追記**:
ツール終了時、その日の枠に**リプ件数だけ**書き込む。

- 今日の見出し（`## 2026-06-01（月）`）が**無ければ**、新しいブロックを追記:
```
## 2026-06-01（月）
- リプ件数: 5
- 手動投稿数: 
- 気づき: 
```
- **あれば**、その下の `- リプ件数:` の行だけ更新

「手動投稿数」「気づき」は空欄のまま残し、Kentaが夜に手書き。曜日は日付から自動計算。既存の手書き運用（5/25〜の枠）は壊さない。

## 10. エラーハンドリング

| 状況 | 挙動 |
|---|---|
| `skin_account_targets.json` が無い/壊れている | エラー文を出して**終了**（巡回できないため） |
| `skin_reply_history.json` が無い | **初回**とみなし、空の履歴で開始（正常） |
| `skin_reply_history.json` が壊れている | 警告を出して**終了**（自動上書きで累計を消さない・Kentaに確認を促す） |
| ブラウザが開けない | URL表示にフォールバック |
| `skin_daily_log.md` が無い/書けない | **警告だけ出してツールは続行**（件数は `skin_reply_history.json` に残る） |
| メニューで無効な入力（変な番号・空） | 「もう一度入力してください」で再入力。落ちない |

ポイント: **記録系（history）は壊れていたら勝手に上書きしない**。リプ累計が消えると困るので安全側に倒す。

## 11. テスト（`skin/tests/test_skin_reply_hunter.py`・`pytest` 対象）

純粋ロジック関数だけをテストする（ファイル操作・ブラウザ・対話は分離して対象外）。

- `suggest_drafts`: 腸活文 → 腸活カテゴリ／ニキビ文 → ニキビ／無該当 → 汎用／複数該当 → 多い方
- `sort_round_list`: 未リプを最優先、あとは古い順
- `record_reply`: リプ記録後に `last_replied_at` と `total_replies` が正しく増える
- ログ追記ロジック: 今日の見出しが無ければ追記、あれば件数行だけ更新

各2〜3ケース。やりすぎない。
