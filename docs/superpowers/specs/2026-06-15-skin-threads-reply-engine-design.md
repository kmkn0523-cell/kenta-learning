# skin Threads リプライ・エンジン 設計書

- 作成日: 2026-06-15
- 対象: skin アカウント（`けん | 大人ニキビを内側から治す`）
- 目的: フォロワー増加の「入口（リーチ）」を完全自動で開ける

---

## 1. 背景と問題

現状の数字（2026-06-15 時点の実測）:

- Threads: 1投稿あたり閲覧数 2〜11、いいね 0。コメント欄の「2」は自分のシーダー由来。
- Instagram: フォロワー 1人、5/18 から1ヶ月フラット。

結論: **フォロワーが増えない原因は投稿の中身ではなく、配信（リーチ）がゼロだから。**
0フォロワーの Threads アカウントが伸びる唯一の確実なレバーは「自分から他人の投稿のコメント欄に出ていく」こと。これを完全自動で回すのが本エンジン。

## 2. 方針の決定事項（ユーザー合意済み）

- 自動化の線引き: **完全自動・凍結リスク許容**
- 1日のリプライ上限: **20〜30件**
- リプライ生成: **テンプレのみ（LLM不使用）**
  - ただし「固定文の連投」=スパム署名なので、後述の薄い文脈化＋強いランダム化で署名を緩和する

## 3. 技術的前提（確認済み）

- 公式 Threads API の `/keyword_search` エンドポイントで、**他人の公開投稿をキーワード検索でき、ヒットした投稿に返信/引用/リポストできる**。
  - 返信上限: 1,000件/24h ローリング（本エンジンの 20〜30件/日 は遥かに下）。
  - 必要スコープ: `threads_keyword_search`（別途申請）。
- 既存 `skin/skin_threads_action.py` に返信送信機能 `post_to_threads(reply_to_id=...)` が既にある → 再利用する。
- 出典:
  - https://developers.facebook.com/docs/threads/keyword-search/
  - https://www.blotato.com/blog/threads-api-pricing

## 4. アプローチ選定

| 案 | 内容 | 判定 |
|---|---|---|
| A | 公式API `keyword_search` で発見→テンプレ返信→`reply_to_id`送信。GitHub Actions cron常駐 | **採用** |
| B | Playwright でブラウザ自動操作 | 却下（セッション維持必須・GH Actionsで回しづらい・凍結リスク最大・保守地獄） |
| C | ハイブリッド（自動＋手動curated） | 却下（中途半端・放置要件に反する） |

A案の根拠: 「ユーザーが手を下さず、GitHub Actions で回り続ける」要件に最も合う。既存 infra 再利用で初心者でも保守可能。ブラウザ不要で凍結リスクも相対的に低い。

## 5. システム設計（A案）

### 5.1 新規スクリプト `skin/skin_reply_engine.py`

`skin/skin_threads_action.py` のパターン（環境変数からトークン読込・対話なし・全自動）を踏襲する。

処理フロー:

1. **キルスイッチ確認**: 環境変数 `SKIN_REPLY_ENGINE_ENABLED` が `true` 以外なら即終了。
2. **発見**: キーワード集合をループし、各キーワードで `/keyword_search` を叩いて直近の公開投稿を取得。
   - キーワード例: `腸活` / `大人ニキビ` / `肌荒れ` / `インナーケア` / `睡眠 肌` / `食事 肌` / `便秘 肌` 等（設定ファイル化）。
3. **選別**（除外フィルタ）:
   - 自分の投稿（`THREADS_USER_ID` 一致）を除外。
   - `skin_reply_history.json` に既出の投稿IDを除外。
   - 同一著者は1日1回まで。
   - 日本語でない・最低文字数未満・露骨な宣伝（URLだらけ等）を除外。
   - 返信不可設定の投稿はスキップ（API応答で判定 or 送信失敗を握り潰す）。
4. **ランキング**: 新しい順＋「早押し」優先（新着ほどコメント上位に出やすく、プロフィールクリックが多い）。
5. **生成（テンプレ）**:
   - `skin/skin_comment_templates.py` のカテゴリ別テンプレを母数として使用。
   - 投稿本文から名詞/キーワードを1語抽出し、テンプレの差し込み枠に入れて軽く文脈化（LLM不要）。
   - 直前に使ったテンプレIDの連続使用を禁止（履歴で管理）。
6. **送信**: `skin_threads_action.post_to_threads(reply_to_id=...)` で返信。
7. **記録**: `skin_reply_history.json` に追記（投稿ID・著者・テンプレID・日時で重複/著者別上限を管理）、`skin_daily_log.md` にその回の件数を追記。
8. **スロットル**:
   - 1日上限 20〜30件（設定値）。
   - 1実行あたり数件（cron 1回で全部出し切らない）。
   - 送信間隔をランダム化（人間らしい揺らぎ）。

### 5.2 設定

- キーワード一覧・1日上限・1実行あたり件数・同一著者上限は、新規 JSON 設定 or 定数で持つ（`skin_account_targets.json` はターゲット手動巡回用なので役割を分ける）。

### 5.3 スケジューリング

- 新規 GitHub Actions ワークフロー（`skin_threads_action.py` 系の cron 設計を流用）。
- 2〜3時間おきに起動、1回あたり数件返信。

### 5.4 必要なシークレット / 事前作業（ユーザーが1回だけ実施）

- Meta ダッシュボードで skin の Threads アプリに `threads_keyword_search` スコープを追加し、トークンを再発行。
- 再発行したトークンを `.env`（`SKIN_*` 系の命名）と GitHub Secrets に登録。

## 6. リスクと対策

- **スパム判定（最大リスク）**: 完全自動＋テンプレは署名が出やすい。対策 = テンプレ母数を大きく持つ／投稿からの1語差し込みで文脈化／同一テンプレ連投禁止／件数を上限の遥か下で運用／送信間隔ランダム化。
- **凍結のronin巻き添え**: skin と ronin は別アカウント（USER_ID・トークン別）なので、アカウント単位の制裁なら ronin は無事。ただし両者は**同一 Meta アプリ配下**のため、アプリ単位の ToS 制裁時は両トークンが死ぬ残存リスクあり。完全隔離は「skin 専用に別 Meta アプリを作る」だが今回はスコープ外（凍結許容方針内で残存リスクとして許容）。
- **API 障害・トークン切れ**: 送信失敗は例外で握り潰し、ログに残してスキップ（既存スクリプトのエラーハンドリング方針に合わせる）。
- **キルスイッチ**: 環境変数 1つで全停止できるようにし、暴走時に即止める。

## 7. スコープ外（YAGNI）

- LLM 生成リプライ（今回はテンプレのみ）。
- Instagram 側（Reels 工場・自動DM）は別エンジンとして別途設計。
- 自動フォロー/いいね/アンフォロー（別レバー、本設計には含めない）。
- skin 専用 Meta アプリの分離。

## 8. 完了の定義

- `SKIN_REPLY_ENGINE_ENABLED=true` のとき、cron 起動で keyword_search → 選別 → テンプレ返信 → 履歴記録 が全自動で動く。
- 1日の返信が上限内に収まり、同一著者・同一投稿への重複返信がない。
- `false` で完全停止できる。
